# オープンカウンタ案件 顧客ブラウザ完結型取り込み 設計書(フェーズ1.5)

**ステータス**: 設計のみ(実装未着手)
**前提となる決定**: 2026-07-11 経営判断により、GビズID資格情報の預託・自動ログインは
恒久的に採用しない。本設計はその制約下でオープンカウンタ(少額)案件を商品に取り込む
唯一の許容ルートである。

## 設計原則(北極星)

**顧客のGビズID資格情報を、いかなる形でも預からない・中継しない・保存しない。**

- ログインは常に顧客本人がブラウザで行う
- 当社システムに流れるのは「顧客が自分の画面で見ている検索結果のデータ」のみ
- 取り込みは常に顧客の明示的なワンクリックで発火する(無人自動化しない)

この構成が規約上クリーンである根拠:
1. GビズID利用規約が禁じるのは「ID/パスワードの第三者への開示・利用させる行為」。
   本設計では資格情報がブラウザの外に一切出ないため、該当しない
2. ブックマークレットは顧客本人が閲覧中のページのDOMを読むだけで、ログイン処理の
   自動化・大量クロール・サーバー負荷を伴わない
3. 社内運用で同方式(bookmarklet_案件取込.js + GAS Web App)の実績が既にある

## 全体フロー

```
[顧客のブラウザ]                         [当社側]

1. 顧客がp-portalにログイン(本人のGビズID)
2. オープンカウンタ案件を通常どおり検索
3. ブックマークレット「📋案件取込」をクリック
   └ 検索結果テーブルのDOMを読み取り
   └ 顧客トークンを添えてGAS Web AppへPOST ──→ 4. GAS: トークン検証
                                                  5. GAS: 顧客専用シートの
                                                     「レコメンド案件」タブに追記
                                                     (取得元=オープンカウンタ取込)
   ←─ 「N件を取り込みました」ポップアップ ────┘
                                               6. 翌朝の日次バッチ:
                                                  取込分を含めてメール配信
                                                  (自動探索分と区別して表示)
```

## コンポーネント設計

### ① 顧客配布ブックマークレット

社内資産 `bookmarklet_案件取込.js` を外販用に改修する。

- **顧客トークンの埋め込み**: ブックマークレットURL内に `intake_token` を含める
  (顧客ごとに生成・配布)。ブックマークレット本体は全顧客共通ロジック
- **読み取り対象**: p-portal検索結果テーブルの表示行のみ(案件名・発注機関・
  締切日時・案件詳細URL等)。ページ遷移・追加リクエストは行わない
- **送信先**: GAS Web App URL(POST、JSON)
- **UX**: 成功時「N件を取り込みました。明朝のレコメンドに反映されます」、
  失敗時「検索結果ページで実行してください」等の明示メッセージ

### ② 顧客トークン(intake_token)

- 顧客マスタに新列 `intake_token` を追加(32字ランダム英数、顧客ごとに一意)
- 用途はGAS Web Appでの顧客特定・認可のみ。**このトークンで読めるものは何もなく、
  書けるのは自分の専用シートの取込行だけ**(漏洩時の被害面を最小化)
- 漏洩・退会時は列の値を再生成/空欄化して即失効。ブックマークレット再配布で復旧
- `setup_customer_sheet.py` に自動生成とブックマークレットURL出力を追加する

### ③ GAS Web App(受け口)

社内で実績のあるGAS Web Appパターンを外販用に新規デプロイ(社内用とは分離)。

処理: `intake_token` → 顧客解決(不一致は404相当で破棄) → ペイロード検証
(1リクエスト最大100件・必須フィールド・文字列長上限) → 顧客の除外キーワード
フィルタ適用 → 案件URLで重複チェック → 顧客専用シートに追記
(ステータス=未確認、**取得元=オープンカウンタ取込**) → 取込件数を返す

- GASは当社管理者アカウントで動かし、各顧客シートへの書き込み権限のみ持つ
- 簡易レート制限(同一トークンから毎分N回まで)をスクリプトプロパティで実装

### ④ bid-service側の変更(小)

- 「レコメンド案件」タブに **「取得元」列を追加**(`自動探索` / `オープンカウンタ取込`)。
  既存の追記ロジック(delivery.py)は自動探索分に `自動探索` を書く
- 日次バッチのメール生成時、前日分の取込行(取得元=オープンカウンタ取込かつ
  ステータス=未確認)をメールの別セクション「お客様取込分」として掲載
- 重複チェックは従来どおり案件URLキーで、取得元をまたいで適用
  (自動探索とブックマークレットで同じ案件が来ても1行)

## 顧客オンボーディング手順(READMEに転記予定)

1. 契約時に顧客専用ブックマークレットURLを発行・案内(トークン入り)
2. 顧客がEdge/Chromeのブックマークバーに登録(1分)
3. 運用: p-portalログイン→オープンカウンタ検索→📋クリック(顧客の手間は1日約1分)

## リスクと限界(正直な記載)

| リスク | 対応 |
|---|---|
| p-portalのHTML構造変更でブックマークレットが壊れる | 取込0件が続いた顧客を管理者ログで検知し、修正版を再配布。「壊れたら直すまで取込停止」は仕様として顧客に事前説明する |
| 顧客がクリックを忘れる/しない | 商品説明で「オープンカウンタは手動取込」と明示。完全自動を約束しない |
| トークンの第三者流出 | 被害面は「他人の取込行が混ざる」まで。再発行で失効可能。GAS側レート制限で悪用を抑止 |
| スマホからは使えない | PC前提の運用として案内 |

## フェーズ分割

- **b-1(MVP)**: ブックマークレット改修+GAS新規デプロイ+取得元列追加。
  工数目安: 小(社内資産の流用が主体)
- **b-2**: 取込分の翌朝メール統合、取込0件アラート、README顧客手順の整備

## やらないこと(再確認)

- ヘッドレスブラウザによるログイン自動化(GビズID規約違反・MFA構造と非互換)
- 顧客の資格情報・証明書・セッションCookieの受領/保存/中継
- ログイン後領域への当社サーバーからの直接アクセス
