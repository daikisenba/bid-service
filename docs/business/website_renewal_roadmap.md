# ホームページ刷新 + 入札秘書LP組み込み ロードマップ

## 現状(実装ベースの把握)

- **技術**: Next.js 14(App Router)/ TypeScript / Tailwind CSS / framer-motion / lottie-react
- **構成**: 実質**単一ページ**(`app/page.tsx` 約897行)の1枚スクロール。
  Hero → 事業内容(3サービス。公共入札支援を含む) → 技術力 → 期待できる効果 →
  代表メッセージ・会社概要 → お問い合わせフォーム(サービス選択のドロップダウン付き)
- **ブランド**: 青系(blue-700中心)、「変化を力に変える伴走者」、信頼×AI効率化
- リポジトリ: ローカル `Desktop/-souki-corporation-website`(Git管理下)

含意: 今は「1ページ・多セクション」構成。入札秘書は**独立したLP(専用URL)**を
持たせるのがSEO上有利(独自キーワードで検索流入を受け、そのページで完結させられる)。
トップは"入り口"、LPは"刈り取り"の役割分担にする。

---

## 大方針: 「文言 → 構成 → デザイン → 実装」の順で手戻りを最小化

実装(HTML/CSS)から入らないこと。**先にテキスト(文言)を確定**すれば、デザインと
実装のやり直しが激減する。幸い文言の核は既に完成している:
- サービスの核: `service_onepager.md`(確定させるべき最重要ドキュメント)
- コピー・LP構成・SEO: `marketing_plan.md`
- 規約: `terms_of_service_draft.md`

---

## 推奨ロードマップ(2段階)

### 【第1段階】最小構成で"営業導線"を最速で立ち上げる(手戻りゼロ・売上直結)

全面刷新を待たず、既存サイトに最小の追加で営業を始められる状態を作る。
デザインの大改修を伴わないため、後戻りが発生しない。

1. **文言の確定**: `service_onepager.md` をレビュー・確定(ここが全ての源)
2. **入札秘書の専用LP(新ルート)を1枚追加**: 例 `app/nyusatsu-hisho/page.tsx`
   (URL: `/nyusatsu-hisho`)。中身は `marketing_plan.md` のLP構成案どおり。
   既存のTailwind・ブランドカラー・アニメ資産を流用して短工数で作る
3. **トップの「事業内容」に導線を追加**: 既存の公共入札支援カードから、または
   新カードとして「入札秘書(月1万円〜)」を置き、LPへリンク
4. **問い合わせフォームの選択肢に追加**: 既存ドロップダウンに
   「入札レコメンド(入札秘書)」を1行追加(数分の改修)
5. **資料(1枚)をPDF化**: `service_onepager.md` をデザイン化し、資料請求・DMで配布

→ この5つで「LPで見せる・フォームで受ける・資料で渡す」の営業3点セットが完成。
   まずここまでを最優先。

### 【第2段階】サイト全面刷新(第1段階で反応を見てから投資)

トライアルの反応・問い合わせ内容を踏まえてから、全面刷新に着手する
(需要が読めない段階で大規模デザイン投資をしない、という既存の判断原則と同じ)。

1. **コンセプト再定義**: 3事業(補助金・入札・営業支援)の見せ方と優先順位を再整理。
   「入札秘書」をプロダクトとして前面に出すか、コンサルの一部に留めるかを決める
2. **サイトマップ確定**: 単一ページから多ページ構成へ。想定:
   - トップ(会社の顔・3事業への入り口)
   - サービス個別ページ(補助金 / 公共入札支援 / 入札秘書LP / 営業支援)
   - 会社概要 / お役立ちコラム(SEO記事置き場) / お問い合わせ
3. **各ページ構成案(ワイヤー)**: テキストレベルで各ページのセクション順を確定
   (`marketing_plan.md`のLP構成が入札秘書ページの雛形)
4. **デザイン方針**: 既存の青系・アニメ資産をどこまで踏襲/刷新するか決める。
   `page.tsx`897行を**セクション単位のコンポーネントに分割リファクタ**してから
   デザイン改修すると、以降の保守が楽になる
5. **実装**: コンポーネント分割 → 各ルート実装 → 共通ヘッダー/フッター/ナビ整備 →
   フォーム改修
6. **SEO・計測の作り込み**: 各ページのtitle/meta/OGP、パンくず、
   構造化データ、アクセス解析、フォーム到達の計測(どのページから問い合わせが
   来たか分かるように)
7. **公開・検証**: 表示崩れ・レスポンシブ・フォーム送信テスト・Lighthouse計測

## 事前に確定すべき事項(実装前の宿題)

これらが未確定だと後工程で手戻りするため、第1段階の前に潰しておく:

- **ホスティング/デプロイ方法**: 現行サイトの公開手段(Vercel / 静的エクスポート /
  レンタルサーバー等)。新ルート追加時のデプロイ手順に直結
- **問い合わせフォームの送信先**: 現在フォーム送信がどこに届く実装か
  (メール送信API・フォームサービス等)。資料請求を受けるなら要確認
- **ドメイン/URL設計**: LPを `/nyusatsu-hisho` にするか `/service/nyusatsu` にするか等
- **資料請求とトライアルのCV(コンバージョン)定義**: 何をゴールにするか
  (資料DL / フォーム送信 / トライアル申込)を決めてから計測を設計

## この順序を勧める理由(まとめ)

- **文言先行**で、デザイン・実装のやり直しを防ぐ(最大の手戻り要因を先に消す)
- **第1段階(最小LP+導線)を全面刷新より先**に置くことで、営業を止めずに始められる。
  刷新の完成を待つ数週間、売上ゼロで待機する事態を避ける
- **反応を見てから全面刷新**に投資する。トライアルの声が、刷新のデザイン・
  コピーの精度を上げる材料になる
